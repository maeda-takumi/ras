# 要件定義書

## 1. 基本情報
- プロジェクト名：RAS（Risk Alert System）
- 作成日：2026/04/08
- 作成者：前田
- 関係者：前田/サポートチーム

## 2. 背景・目的
- 背景：返金対応の数が増えてきているため、削減したいが分析する環境が整っていない
- 目的：顧客からの問い合わせを分析し、返金につながりそうな顧客を早めに察知しクロージングを行う
- 解決したい課題：これまでの返金対応になった顧客とのやりとりや特徴をデータ化し、返金になりそうな顧客を察知するロジック構築

## 3. 対象範囲
- 対象：
　チャットの自動取得（LMessage）
　返金対応者についての学習
　ステータス管理
　チャット履歴の表示
　対応管理
　ステータスログの記録
　対応アドバイス

- 対象外：
　LMessageとの連携
　顧客への自動返信対応
　本システムからの返信
　LMessage以外への対応

## 4. 利用者
- 利用者種別：サポートチーム
- 想定人数：1～5
- 利用シーン：サポート業務

## 5. 業務フロー
- 現状フロー：
　1.LMessageからチャットの履歴を全取得
　2.ChatGPTを使い返金対応者、クレーマーを抽出
　3.抽出者のチャット分析
　4.スプレッドシートへ記録

- 新フロー：
　1.LMessageからチャットの履歴を1日分取得
　2.AIを使った自動分類分け
　3.利用者がAIがつけた分類分けをみて、対応

## 7. 非機能要件
- 性能：
　　利用者はデータの取得分析について、考慮する必要が内容にする
　　スクレイピング失敗による、処理停止が起きないうにする
　　ユーザの増加による処理時間の増加が極力起きないようにする
　　機械学習による精度の上昇が期待できるようにする　　

- セキュリティ：
　　ログイン制による、ユーザ情報の保護を行う

- 可用性：
　　サポートチームが社内社外関係なく使用できるようにする
　　取得システムが停止してもサポートチームが動作再開できるようにする
　　定時実行以外にも手動実行可能にする
　　プロンプトやログイン者の管理を行う
　　ログイン情報の保存処理を簡易的に行うように
　　ポーリング中でもログイン情報の保存ができるように　　

- 保守性：
　　スクレイピング処理、AI判定処理、表示処理は分離して実装する
　　
- 権限管理：
　一般、管理者と分け、一般はWeb閲覧のみ
　管理者は設定値の変更、およびログインユーザの変更が可能
